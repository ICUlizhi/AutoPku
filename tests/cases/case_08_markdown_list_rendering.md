## 问题描述

用户的核心反馈是：**PDF 中大量 Markdown 列表项（`- ` 开头）没有正确渲染为带项目符号的分点列表**，而是显示为连续的普通文本，`-` 被当成了普通连字符。

例如 `**陈述**:` 段落后面的列表项：

```markdown
**陈述**:
- **个别财务报表**：单个公司编制的母公司自身的财务报表。
- **合并财务报表**：把母子公司形成的企业集团...
```

在 PDF 中被渲染为：

```
陈述: - 个别财务报表：... - 合并财务报表：...
```

完全失去了列表结构。

- **Agent B**: 扫描列表项后未缩进的块级元素 → 仅报告 handout1 中 1 处 `$$` 公式块

**结果**：修复了 handout1 公式空行、handout6 两处嵌套列表缩进，但用户反馈"还是有问题"。

### 第二轮：主 Agent 亲自诊断

主 Agent 通过 `pandoc --to=latex` 检查 LaTeX 中间输出，发现关键线索：

```bash
$ pandoc notes/handout1_课程导论与框架.md --to=latex
# 定义 1.1 的列表完全被吞掉，只有一个 itemize 环境
# 定义 1.5 被渲染为：
# \textbf{陈述}: - \textbf{基本公式}: ... - \textbf{加权平均净资产}: ...
```

进而通过最小复现实验锁定根因：

```bash
$ cat > test.md << 'EOF'
**陈述**:
- **项目1**：内容1
- **项目2**：内容2
EOF

$ pandoc test.md --to=latex
\textbf{陈述}: - \textbf{项目1}：内容1 - \textbf{项目2}：内容2
# 列表完全失效！

$ cat > test2.md << 'EOF'
**陈述**：

- **项目1**：内容1
- **项目2**：内容2
EOF

$ pandoc test2.md --to=latex
\textbf{陈述}：

\begin{itemize}
\tightlist
\item
  \textbf{项目1}：内容1
\item
  \textbf{项目2}：内容2
\end{itemize}
# 列表正常！
```

**根因确认**：**pandoc 的 Markdown 解析规则——段落后面直接跟 `- ` 列表项（中间无空行）时，列表完全失效。**

## 踩坑与经验

### 1. 子 Agent 的第一次扫描没有找到真正的问题

子 Agent 的扫描逻辑聚焦于"列表项后面跟了块级元素"，但实际问题是"**段落后面跟了列表项**"（方向相反）。这导致第一轮修复仅处理了边缘情况，没有触及核心问题。

### 2. 直觉猜测 vs 中间格式诊断

- ❌ 错误路径：凭 Markdown 肉眼检查猜测问题（缩进？callout？代码块？）
- ✅ 正确路径：通过 `pandoc --to=latex` 或 `pandoc --to=json` 查看中间输出，直接观察 AST/LaTeX 中列表环境是否存在

### 3. pandoc 的 Markdown 解析陷阱

**pandoc 的列表识别规则**：
- 标题（`#` / `##`）后面直接跟 `- ` → ✅ 正常识别为列表
- 段落（普通文本行）后面直接跟 `- ` → ❌ **列表失效**，`-` 被当作普通连字符
- 必须在段落和列表项之间插入**至少一个空行**

### 4. Agent Team 的分阶段迭代策略

本次 session 展示了 Agent Team 协作的最佳实践：

```
用户反馈 → 第一轮扫描（发现边缘问题）→ 用户再次反馈 → 
主 Agent 亲自诊断（锁定根因）→ 第二轮大规模修复 → 验证
```

当子 Agent 的扫描结果与用户反馈不匹配时，主 Agent 应该亲自介入做最小复现实验，而不是继续让子 Agent 在错误方向上迭代。

## 测试价值

此案例可用于验证 `write-notes` 的 Markdown → PDF 渲染管道：

1. **列表渲染测试**：验证 `**陈述**:\n- 项目1` 这类常见结构是否被正确渲染为 `itemize` 环境
2. **批量修复测试**：验证 Agent Team 能否正确识别并修复跨多个文件的 Markdown 语法问题
3. **pandoc 兼容性测试**：确保生成的 Markdown 符合 pandoc 的解析规范（段落与列表之间需要空行）
4. **回归测试**：在修改 `write-notes` 的 prompt 或模板后，检查新生成的笔记是否避免了此类问题
